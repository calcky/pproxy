## memif failure detail

Before the ARP fix, the memif scenario completed deployment:

- DPDK WAN devices were restored to the kernel driver on both leaves.
- VPP sidecars started on both leaves.
- VPP created `memif1/0` and `host-ens2`, and installed L2 xconnect in both directions.
- pproxy processes were started on both leaves.

The path did not become reachable within the matrix connectivity window because router neighbor entries for leaf WAN IPs stayed unresolved:

```text
==> waiting for path client1 -> 192.168.1.2 (up to 60s) ...
!!! ping failed after 60s
```

Manual static ARP on the router immediately restored connectivity, so `deploy.sh` now writes the same static ARP entries in the memif branch. After that change, the full matrix completed and `udp_memif` passed both P1 and P10 runs.
